# PRD — 모임 이벤트 관리 웹 (MVP)

> **한 줄 정의:** 일회성 모임을 여는 비전문 주최자가, 참여자에게 **링크 하나**를 보내 참석 여부를 모으고 공지를 전달하는 가장 빠른 도구.

- **문서 버전:** v0.1 (MVP)
- **작성일:** 2026-06-27
- **상태:** 기획 확정 → 개발 착수 대기
- **기술 전제:** Next.js 15 App Router + Supabase (본 레포 스택)

---

## 1. 배경 & 문제 정의

수영·헬스·동호회·번개 모임을 여는 주최자는 모임 한 번을 위해 공지, 참여자 집계, 카풀, 정산 등 여러 일을 떠안는다. 대부분은 **카카오톡 단톡방 + 엑셀 + 메모**로 흩어져 처리되며, 그 결과:

- **참석자 수를 끝까지 모른다.** "갈게요"가 단톡방에 흘러가 집계가 안 된다 → 장소·식당 예약 인원을 못 정한다.
- **공지가 묻힌다.** 중요한 공지(시간 변경, 준비물)가 수다 사이에 묻혀 "못 봤어요"가 반복된다.
- **명단을 손으로 센다.** 누가 온다 했는지 매번 스크롤하며 손으로 카운트한다.

### 문제의 핵심

> 주최자의 시간이 **참여자 응답을 모으고 세는 일**에 가장 많이 쓰인다. MVP는 **이 한 가지**를 자동화한다.

### 왜 지금, 왜 이 형태인가

- 기존 도구(구글폼, 네이버폼)는 "응답 수집"은 되지만 **공지·실시간 집계·참여자 관점 UX**가 없다.
- 단톡방은 소통은 되지만 **구조화된 집계가 없다.**
- 둘 사이의 빈틈 — *"링크 보내고 → 자동 집계 + 공지"* — 를 정확히 메운다.

## 2. 목표 & 성공 지표

### 2.1 제품 목표 (MVP가 증명할 가설)

> **가설:** 주최자는 "단톡방 + 엑셀"을 버리고, 이벤트당 링크 하나로 참여자를 관리할 의향이 있다.

### 2.2 성공 지표 (North Star + 보조)

| 구분 | 지표 | MVP 목표치 | 측정 방법 |
| --- | --- | --- | --- |
| **North Star** | 이벤트당 RSVP 응답 수 | 평균 8명 이상/이벤트 | RSVP 레코드 집계 |
| 활성화 | 이벤트 생성 후 24h 내 링크 공유율 | 70% 이상 | 링크 최초 조회 발생 비율 |
| 핵심 가치 | 참여자 링크 진입 → RSVP 완료 전환율 | 60% 이상 | 페이지뷰 대비 RSVP |
| 리텐션 | 2번째 이벤트를 만든 주최자 비율 | 30% 이상 | 주최자별 이벤트 수 |

### 2.3 명시적 비목표 (Non-Goals)

- 정산·송금 (카카오페이/토스 연동) — **로드맵 v1.1**
- 카풀·좌석 매칭 — **로드맵 v1.2**
- 동호회/클럽 회원·회비 관리 (장기 운영) — **타깃 아님**
- 결제·유료 티켓팅
- 모바일 네이티브 앱 (모바일 웹으로 충분)
- 실시간 채팅 (단톡방을 대체하지 않음. 공지는 단방향)

## 3. 타깃 사용자 & 페르소나

### 주 페르소나 — "번개 주최자 지민" (일회성 이벤트 기획자)

- 30대 직장인. 이번 주말 등산 번개를 카톡방에 올렸다.
- "갈게요" 12개, "고민중" 5개가 대화에 섞여 있어 **정확히 몇 명인지 모른다.**
- 식당 예약을 위해 인원 확정이 급하다.
- **니즈:** 가입·앱 설치 없이, 링크 하나 던지고 실시간으로 인원이 세지는 것.

### 부 페르소나 — "참여자 현우"

- 지민이 보낸 링크를 카톡에서 받는다.
- **가입하기 싫다.** 30초 안에 "참석"만 누르고 끝내고 싶다.
- **니즈:** 로그인 없이, 이름만 적고 참석/불참/미정을 빠르게 응답.

### 설계 원칙 (페르소나에서 도출)

1. **주최자만 가입.** 참여자는 공유 링크로 가입 없이 RSVP. (진입 장벽 최소화)
2. **모바일 웹 우선.** 모든 상호작용은 카톡 인앱 브라우저/모바일에서 한 손으로.
3. **링크가 곧 제품.** 핵심 사용 흐름은 "링크 생성 → 공유 → 응답 수집".

## 4. 사용자 시나리오 (핵심 플로우)

### 4.1 주최자 플로우

```
가입/로그인 → [이벤트 생성] → 제목·일시·장소·정원 입력
           → 공유 링크 발급 → 카톡으로 링크 전송
           → 대시보드에서 참석/불참/미정 실시간 확인
           → 공지 작성(시간 변경 등) → 링크 페이지에 노출
```

### 4.2 참여자 플로우

```
카톡에서 링크 클릭 → 이벤트 상세 페이지(가입 불필요)
                 → 이름 입력 + [참석] / [불참] / [미정] 선택
                 → "응답 완료" → 현재 참석자 수 확인 + 공지 열람
                 → (선택) 같은 링크로 재방문해 응답 변경
```

## 5. 기능 명세 (MVP 범위)

우선순위: **P0 = MVP 필수**, **P1 = MVP 포함되면 좋음**, **P2 = 로드맵**

### 5.1 주최자 인증 (P0)

- 본 레포의 Supabase 쿠키 기반 SSR 인증 사용 (이메일 또는 Google OAuth).
- 주최자만 계정 필요. 회원가입 시 `profiles` 자동 생성(기존 트리거).

### 5.2 이벤트 관리 (P0)

| 기능 | 설명 | 우선순위 |
| --- | --- | --- |
| 이벤트 생성 | 제목, 시작·종료 일시, 장소(텍스트), 정원(선택), 설명 | P0 |
| 이벤트 수정/삭제 | 본인 이벤트만 (RLS) | P0 |
| 공유 링크 발급 | 추측 불가능한 slug/토큰 기반 공개 URL | P0 |
| 이벤트 목록 | 주최자 대시보드에 본인 이벤트 카드 | P0 |
| 정원 마감 | 정원 도달 시 참석 응답 차단 + "대기" 처리 | P1 |

### 5.3 RSVP — 참여자 응답 (P0, 핵심)

| 기능 | 설명 | 우선순위 |
| --- | --- | --- |
| 가입 없는 응답 | 이름 입력 + 참석/불참/미정 3종 | P0 |
| 응답 변경 | 같은 링크 재방문 시 수정 (브라우저 토큰/쿠키로 식별) | P0 |
| 실시간 집계 | 참석/불참/미정 수 카운트 표시. 참석은 **응답자 수**와 **총 참석 인원(동반 합산)** 둘 다 표시 | P0 |
| 참석자 명단 | 주최자 대시보드 + 공개 페이지에서 응답자 이름·상태 목록 표시 | P0 |
| 일행 동반 인원 | "+N명 동반" 입력 (등산/식사 인원 산정용). 집계는 인원 합산 기준 | P0 |

### 5.4 공지 (P0)

| 기능 | 설명 | 우선순위 |
| --- | --- | --- |
| 공지 작성 | 주최자가 이벤트에 공지 텍스트 추가 | P0 |
| 공지 노출 | 참여자 링크 페이지 상단에 최신 공지 표시 | P0 |
| 공지 이력 | 여러 공지를 시간순 표시 | P1 |

### 5.5 로드맵 (P2 — MVP 비범위, 명시적 기록)

- **정산 (v1.1):** 1/N 계산, 송금 링크(토스/카카오페이) 첨부, 입금 체크.
- **카풀 (v1.2):** 차량 등록, 좌석 수, 탑승자 매칭, 출발지.
- **알림:** 이벤트 임박 리마인더(카카오 알림톡/이메일).
- **반복 이벤트:** 정기 모임용 템플릿 복제.

## 6. 데이터 모델 (개념 설계)

> 본 레포의 Supabase 컨벤션(RLS, `profiles` 트리거)을 따른다. 아래는 신규 테이블의 개념 스키마.

### `events`
| 컬럼 | 타입 | 비고 |
| --- | --- | --- |
| `id` | uuid (PK) | |
| `host_id` | uuid (FK → profiles) | 주최자 |
| `slug` | text (unique) | 공개 링크용 추측 불가 토큰 |
| `title` | text | |
| `starts_at` | timestamptz | 모임 시작 일시 |
| `ends_at` | timestamptz | 모임 종료 일시 |
| `location` | text | 장소(텍스트) |
| `capacity` | int (nullable) | 정원, null = 무제한 |
| `description` | text (nullable) | |
| `created_at` / `updated_at` | timestamptz | 트리거로 갱신 |

### `rsvps`
| 컬럼 | 타입 | 비고 |
| --- | --- | --- |
| `id` | uuid (PK) | |
| `event_id` | uuid (FK → events) | |
| `guest_token` | text | 가입 없는 참여자 식별(쿠키 저장) |
| `name` | text | 응답자 이름 |
| `status` | enum(`going`,`not_going`,`maybe`) | |
| `party_size` | int (default 1) | 본인 포함 참석 인원 (동반 합산) |
| `created_at` / `updated_at` | timestamptz | |

### `announcements`
| 컬럼 | 타입 | 비고 |
| --- | --- | --- |
| `id` | uuid (PK) | |
| `event_id` | uuid (FK → events) | |
| `body` | text | 공지 내용 |
| `created_at` | timestamptz | |

### RLS 정책 방향

- `events` 쓰기/수정/삭제: `host_id = auth.uid()` (주최자 본인만).
- `events` 공개 읽기: slug를 아는 누구나(가입 불필요) → 공개 읽기는 **slug 기반 조회 전용 경로/RPC**로 노출하고, 직접 테이블 SELECT는 제한.
- `rsvps` 쓰기: 가입 안 한 참여자도 가능해야 하므로 **slug 검증을 거치는 Server Action/RPC**를 통해서만 insert/update. 주최자는 본인 이벤트의 rsvp 전체 조회 가능.
- `announcements` 쓰기: 주최자 본인. 읽기: 해당 이벤트 slug 보유자.

> ⚠️ 핵심 보안 과제: "가입 없는 쓰기"를 허용하면서 스팸·도배를 막아야 한다. MVP에서는 slug 기반 접근 + 간단한 rate limit으로 시작하고, 어뷰징 발생 시 캡차/토큰 강화.

## 7. 화면 구성 (와이어 수준)

1. **랜딩/로그인** — 주최자 가입·로그인 (기존 스타터 UI 활용).
2. **주최자 대시보드** (`/dashboard`) — 내 이벤트 카드 목록 + "새 이벤트" 버튼.
3. **이벤트 생성/수정** (`/events/new`, `/events/[id]/edit`) — 폼.
4. **주최자 이벤트 상세** (`/events/[id]`) — 집계 요약(참석 N/불참 N/미정 N), 명단, 공지 작성, 공유 링크 복사.
5. **공개 RSVP 페이지** (`/e/[slug]`) — 가입 없이 접근. 이벤트 정보(시작·종료 시각) + 최신 공지 + RSVP 폼(이름·참석여부·동반 인원) + 현재 참석 수(응답자/총인원) + **참석자 이름 명단**.

## 8. 범위 & 일정 (MVP 절단선)

### In Scope (MVP)
- 주최자 인증, 이벤트 CRUD, 공유 링크, 가입 없는 RSVP, 실시간 집계, 공지.

### Out of Scope (로드맵)
- 정산, 카풀, 알림, 반복 이벤트, 채팅.

### 대략적 마일스톤 (제안)

| 단계 | 범위 | 산출물 |
| --- | --- | --- |
| M1 | 데이터 모델 + 마이그레이션 + RLS | events/rsvps/announcements 테이블 |
| M2 | 주최자 이벤트 CRUD + 대시보드 | 인증된 주최자 흐름 완성 |
| M3 | 공개 RSVP 페이지 + 집계 | 참여자 핵심 가치 완성 (데모 가능) |
| M4 | 공지 + 공유 UX 다듬기 | MVP 출시 |

## 9. 리스크 & 열린 질문

| 리스크 / 질문 | 영향 | 대응 |
| --- | --- | --- |
| 가입 없는 RSVP의 어뷰징·중복 | 데이터 신뢰도 | slug 검증 + rate limit, 필요 시 토큰 강화 |
| 참여자 식별을 쿠키로만 하면 기기 변경 시 응답 수정 불가 | UX | MVP는 허용 한계로 두고, 재방문 시 같은 기기 안내 |
| 카톡 인앱 브라우저 호환성 | 진입 전환율 | 모바일 웹 우선 테스트 필수 |
| "공지 단방향"이 단톡방을 대체 못함 | 채택률 | 의도된 범위. 집계+공지로 충분한지 MVP로 검증 |
| 공개 명단에 실명 노출 (확정 반영) | 프라이버시 | 참여자가 입력한 이름이 공개됨을 RSVP 폼에 명시 고지. 닉네임 입력 허용 |

### 확정된 결정 (이전 열린 질문)
- **종료 시각:** 필요. `events.ends_at` 추가, 생성 폼에 시작·종료 입력. ✅
- **동반 인원(`party_size`):** P0로 승격. 집계는 인원 합산 기준. ✅
- **공개 명단:** 참여자에게도 **참석자 이름 명단**을 표시. (수만 보여주지 않음) ✅

## 10. 부록 — 한 줄 요약

> **"링크 하나로 끝내는 모임 참석 집계."** 주최자는 가입하고 이벤트를 만들어 링크를 뿌리고, 참여자는 가입 없이 이름과 참석 여부만 누른다. 정산·카풀은 다음 버전.
